# CLAUDE.md — Next.js 15 + SQLite SaaS Project

## Stack & Versions

| Layer | Choice | Reason |
|-------|--------|--------|
| Framework | Next.js 15 (App Router) | RSC-first, stable server actions, PPR |
| Database | SQLite via better-sqlite3 | Zero infra, fast CI, great for single-tenant SaaS |
| ORM / Query | Drizzle ORM | Type-safe, SQL-like syntax, lightweight (no magic) |
| Auth | lucia-auth v3 | Cookie-based, works with SQLite, no external dep |
| UI | Tailwind CSS 4 + shadcn/ui | Utility-first + accessible components |
| Forms | react-hook-form + zod | Performant, type-safe validation |
| Deployment | Docker + single VPS (or Fly.io) | SQLite is a single-file DB — pair with a single VM |
| Node | >= 20 LTS | Stable, native fetch, .env file support |

**Do NOT** add Prisma (too heavy for SQLite), tRPC (overkill for server actions), or any JS runtime other than Node (SQLite bindings don't work in Edge/Bun/Deno).

## Folder Structure

```
src/
├── app/                    # Next.js App Router pages
│   ├── (auth)/             # Auth routes group (login, register)
│   ├── (dashboard)/        # Authenticated routes group
│   ├── api/                # Route handlers (webhooks, external)
│   └── layout.tsx
├── components/             # Shared React components
│   ├── ui/                 # shadcn/ui primitives (button, input, etc.)
│   ├── forms/              # Form components tied to specific schemas
│   └── layout/             # Sidebar, header, nav wrappers
├── db/
│   ├── schema/             # Drizzle schema files (one per domain)
│   │   ├── users.ts
│   │   ├── projects.ts
│   │   └── ...
│   ├── migrations/         # Auto-generated by `drizzle-kit`
│   ├── index.ts            # DB client singleton
│   └── seed.ts             # Dev seed data
├── lib/
│   ├── auth.ts             # Lucia auth config
│   ├── email.ts            # Email sending (Resend / Plunk)
│   └── utils.ts            # Pure helper functions
├── actions/                # Server actions (one file per domain)
│   ├── auth.actions.ts
│   ├── projects.actions.ts
│   └── ...
├── hooks/                  # Client-side React hooks
└── styles/                 # Global CSS / Tailwind config
```

**Why flat `actions/` instead of co-located with pages?** Server actions are RPC-like. Co-locating them with pages encourages coupling. Keeping them flat makes it easy to reuse across routes and test independently.

**Why `db/schema/` per-domain instead of one file?** A single `schema.ts` becomes a 1000+ line monster fast. One file per domain keeps dependencies explicit and diffs readable.

## Naming Conventions

| Type | Convention | Example |
|------|-----------|---------|
| Components | PascalCase | `UserAvatar.tsx` |
| Server Actions | kebab-case.actions.ts | `team.actions.ts` |
| DB Schemas | PascalCase per-table | `users.ts` → `users` table |
| Hooks | camelCase with `use` prefix | `useCurrentUser.ts` |
| Lib files | camelCase | `sendEmail.ts` |
| Folders | kebab-case | `user-settings/` |
| Types/Interfaces | PascalCase, `Type` suffix for types | `UserProfileType` |

**No index.ts barrel exports.** They hide circular dependencies and confuse tree-shaking. Import directly: `import { Button } from '@/components/ui/button'`.

## SQL / Migration Conventions

### Rules

1. **Every migration must be reversible.** Drizzle generates both `up` and `down`. Review the down migration before committing.
2. **Never edit a migration after it's merged.** If you need to change the schema, write a new migration.
3. **Timestamps are TEXT (ISO 8601).** SQLite has no native datetime type. Storing ISO strings (`2026-06-07T10:00:00Z`) is sortable, human-readable, and timezone-safe. Add `NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))`.
4. **Use INTEGER for booleans.** SQLite has no native boolean. `1` / `0` with a CHECK constraint.
5. **Every table gets:**
   ```ts
   id: text().primaryKey().$defaultFn(() => nanoid(21)),
   createdAt: text().notNull().default(sql`(strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))`),
   updatedAt: text().notNull().default(sql`(strftime('%Y-%m-%dT%H:%M:%SZ', 'now'))`),
   ```
6. **Use TEXT primary keys (nanoid).** Auto-increment integers leak record counts. UUIDs are slow. nanoid(21) is compact, sortable, URL-safe.
7. **Always add foreign keys explicitly.** Drizzle doesn't enforce FKs by default. Add `.references(() => users.id)` and `PRAGMA foreign_keys = ON` on connection.

### Dev Commands

```bash
# Generate migration after schema change
pnpm db:generate

# Apply migrations
pnpm db:migrate

# Open SQLite console
pnpm db:studio

# Reset DB (drop all tables, re-run all migrations)
pnpm db:reset
```

## Component Patterns

### Server Components First
- Default to Server Components. Only add `'use client'` when you need interactivity (hooks, event handlers, browser APIs).
- Data fetching happens in Server Components. Pass data down as props.

### The "Page → Feature → UI" stack
```
page.tsx          → imports feature components
features/         → orchestrates data + UI logic
components/ui/    → pure presentational (no data fetching)
```

### Form Pattern
```tsx
// app/(dashboard)/settings/page.tsx — Server Component
import { updateName } from '@/actions/settings.actions'
import { NameForm } from '@/components/forms/name-form'

export default function SettingsPage() {
  return <NameForm action={updateName} />
}

// components/forms/name-form.tsx — Client Component
'use client'
export function NameForm({ action }: { action: (formData: FormData) => Promise<ActionResult> }) {
  // useActionState from react-dom
}
```

**Never** use `fetch()` in a Client Component for your own API. Use Server Actions. They avoid the HTTP roundtrip and work with Progressive Enhancement.

## What We Don't Do (And Why)

| Anti-pattern | Why we avoid it |
|-------------|----------------|
| Redux / Zustand / Jotai | Server Components + URL search params handle 95% of state. The other 5% is React Context. |
| API routes for internal data | Server Actions exist. Extra HTTP hops = slower UX. |
| Edge Runtime | SQLite bindings don't work. Pin Node.js Runtime. |
| GraphQL | Over-engineered for a SQLite-backed app. Drizzle gives you type-safe queries without a query language. |
| `'use client'` on layouts | Forces the entire layout tree to be client-side. Layouts should be Server Components. |
| Catch-all error boundaries on every page | One root error boundary + per-feature boundaries where it matters (forms, payments). |
| CSS-in-JS (styled-components, emotion) | Runtime cost. Tailwind + CSS Modules cover everything. |
| Commit messages like "fix" or "update" | Use conventional commits. `feat:`, `fix:`, `chore:`, `refactor:`. Enables auto-changelog. |
| Mixing `pnpm` and `npm` | Stick to `pnpm`. Lockfile consistency matters. |

## Quick Start

```bash
pnpm install
cp .env.example .env
pnpm db:migrate
pnpm db:seed
pnpm dev
```

> **Principle:** A new developer should be able to go from `git clone` to `localhost:3000` with data in under 2 minutes. If you find yourself adding a step, consider whether it's really necessary.
